\chapter{Simulator}
The simulator general objective is to have a instrument to validate our ideas and results in a experimental way. In particular we want obtain experimentally for each task, the distribution of job execution time.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\section{Taskset Model}
\label{sec:simulator_task_model}
In this section we define the taskset model, i.e., how the tasks in the taskset can be defined and which are their non functional characteristics.
From now on we suppose that:
\begin{itemize}
    \item We suppose to have a single core CPU, which implies that we are working with a concurrent problob.
    \item Each task is independent from the others. This means that there aren't shared resources among two tasks.
    \item Each task have a single elementary execution unit, i.e. each task have only one chunk.
    \item The deadline og each task is known at priory.
    \item The task execution time is defined probabilistically, i.e. the execution time of each job of a task is sampled from some distribution. These distribution can include also the deterministic one.
    \item The jobs execution time are i.i.d. (i.e., independent and identically distributed). This means that only one distribution represents the job execution time pattern of a given task.
    \item Task can be both periodic and sporadic (i.e. the releases between two consecutive job of the same task are separated from a minimum inter-arrival time). For the sporadic task, each job releasing time can be sampled from some distribution.
\end{itemize}

\section{Input and Output}
The simulator input are the following:
\begin{itemize}
    \item A fixed priority scheduling algorithm. As we'll see in Section5\ref{sec:simulator_implementation} the scheduling algorithm implemented is Deadline Monotonic: we can handle periodic and aperiodic tasks with a fixed priority.
    \item The taskset, characterized from the assumptions in Section~\ref{sec:simulator_task_model}.
    \item The duration of the simulation.
    \item The type of logging. By default no logging are performed.
\end{itemize}

\subsection{Logging}
To test the scheduler algorithm implementation and to have an evidence of the system behaviour, the simulator can produces a file, named \textit{trace.log}, containing a sequence of \textit{<time, event>}, where \textit{event} can be: the release of a job, the execution and the completition of a job, the task preemption and the deadline miss event.

\subsection{Deadline Miss}
The system has obviusually the capabilities to identifies a deadline miss by a task and can also identifies the job that is responsable of this violation. A violation implies the abort of whole simulation.

This check is implemented by a single function (i.e., \texttt{checkDeadlines()} defined in \texttt{Scheduler.java}) to have a single poit of truth that called after each time advancement or each relevant event.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\section{Time}
\label{subsec:simulator_time}
This kind of simulation must be totally deterministic; for this reason all timing aspects have been handle in a discrete way using the \texttt{Duration} Java type. In this way we can handle the complexity of small duration with a high abstraction level.

We assume that the time is taken in input and returned in output in millisecond. During the execution, to obtain the maximum precision, it's elaborated in nanoseconds.

\section{Implementation}
\label{sec:simulator_implementation}
In Figure~\ref{fig:simulator_class_diagram} is shown the class diagram of the simulator. In the following we present the core entities that compose the system.

\begin{itemize}
    \item [\textbf{Client}] Is the Main of the Java systems. It is responsable to allocate the taskset and the scheduler and call \texttt{analyse()} on the scheduler.
    \item[\textbf{TaskSet}] Is a simple set of all task.
    \item[\textbf{Task}] Represent a task. It model only the static specification of the task: the period, the execution time and the deadline; the first two are a Sirio \texttt{Sampler} that represent the distribution from wich they sample the true duration; the last is a \texttt{Duration} objects, where the time must be expressed in milliseconds. Internally it also maintains a \texttt{jobCounter} to uniquely identify its jobs.
    \item[\textbf{Job}] Represent a task instance. While a Task contain only the task specifications, a Job contains the active state: the absolute deadline, the remaining execution time, the information about its completeness and the time of completition. It has a method to handle the simulation of its execution.
    \item[\textbf{Scheduler}] Represent the heart of the simulation, that use the taskset for the scheduling simulation. It's responsable to initilizate the system clock with a \texttt{MyClock} object and handle the system logging and the duration of simulation.
    \item[\textbf{FixedPriorityScheduler}] Represent the abstraction for all fixed priority scheduling algorithms. All its subclass must only impement the assign priority method.
    \item[\textbf{DMScheduler}] Is the specialization of the \texttt{Scheduler} that implement the priority assignment of Deadline Monotonic scheduling algorithm.
    \item[\textbf{Event}] Is the abstract class the represent the relevant event of the system; each event is defined by the time of its occurence . The idea is that the time can go forware untill it reach an instance of this class. The two implementation are: \texttt{ReleaseEvent} that modeling the job release event; \texttt{DeadlineEvent} that modeling the reaching of the job deadline.
\end{itemize}

\subsection{Utilities}
if we observe the Figure~\ref{fig:simulator_class_diagram} we have to describe the systems utilities.
\begin{itemize}
    \item [\textbf{ConstantSampler}] A specialization of the Sirio \texttt{Sampler} interface that always returns a constant value, used for deterministic parameters.
    \item [\textbf{MyClock}] Represents the global clock of the system. Implements the method to go at specified time, to go forward in time by a specified duration and to print the current time.
    \item [\textbf{MyLogger}] An interface that abstracts the logging operations.
    \item [\textbf{TraceLogger}] Implementation of \texttt{TraceLogger} that is responsable to log in a file named \textit{trace.log} with a personalized format.
    \item [\textbf{NoLogger}] Implementation of \texttt{TraceLogger} that doesn't perform any logging operation. It's the deafult logger if Main pass no logger.
    \item [\textbf{DeadlineMissedException}] A custom exception thrown when a task's job exceeds its deadline.
\end{itemize}

\subsection{Flow}
Let's analyse, thanks also to the Figure~\ref{fig:simulator_activity_diagram},the scheduler flow. This follow the steps below:
\begin{enumerate}
    \item Reset the state of the whole system. This allow the client (i.e., the \texttt{Main.java}) to call more and more time \texttt{analyze()} to obstain consecutive scheduling.
    \item Release the first job of all tasks.
    \item Iterate through all events untill we found a next event with time higher than the simulation duration.
    \begin{enumerate}
        \item Take and remove the first event from the event queue and calculate the available time from now to this current event.
        \item Distribute the available time among all ready jobs according to their priority.
        \begin{enumerate}
            \item Take the highest priority job from the ready jobs queque.
            \item Evaluete if the last job executed must be preempted by this new highest priority job.
            \item Give to the job the available time that it can exploit and execute the job.
            \item Advace with the time clock by the real job execution time and check for deadline miss.
            \item Calculate the new available time.
            \item If the job hasn't finished its work we insert another time in the ready job list for the next iteration.
        \end{enumerate}
        \item Carry the logical clock current time to the our event time. Then check the deadline miss for all acrive jobs; with this we ensure that the symulator state is correct and we can go on.
        \item Consider all event that have the same time of our current event. We have to deal with them all toogher. For these events that represent a new release, we release a new job of related task. The release behaviour is: create a new job and add it to all related structures; add the its deadline event to the event queue; add the new release task time to the event queque.
    \end{enumerate}
    \item It the current time is less lower than the simulation duation execute the simulation for the remaining time.
\end{enumerate}
