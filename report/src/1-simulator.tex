\chapter{Simulator}
The simulator's general objective is to have an instrument to validate our ideas and results in an empirical way. In particular we want to obtain experimentally for each task, the distribution of job execution time.

This kind of simulation must be totally deterministic; for this reason all timing aspects have been handled in a discrete way using the \texttt{Duration} Java type. In this way we can handle the complexity of small durations with a high abstraction level. We assume that the time is taken as input and returned as output in milliseconds. During the execution, to obtain the maximum precision, it's processed in nanoseconds.

\section{Taskset Model}
\label{sec:simulator_task_model}
In this section we define the taskset model, i.e., how the tasks in the taskset can be defined and what are their non-functional characteristics.
From now on we assume that:
\begin{itemize}
    \item We assume we have a single core CPU, which implies that we are working with a concurrent problem.
    \item For tasks:
    \begin{itemize}
        \item Each task is independent of the others. This means that there aren't shared resources among two tasks.
        \item Tasks can be both periodic and sporadic (i.e. the releases between two consecutive jobs of the same task are separated by a minimum inter-arrival time). For the sporadic task, each job release time can be sampled from some distribution.
        \item Each task has a single elementary execution unit, i.e. each task has only one chunk.
        \item The task execution time is defined probabilistically, i.e. the execution time of each job of a task is sampled from some distribution. These distributions can include also the deterministic ones.
        \item The deadline of each task is known a priori.
        \item The first release time is by deafult at time 0, but it ca be sampled from some distribution as well.
    \end{itemize}
    \item The job execution times are i.i.d. (i.e., independent and identically distributed). This means that only one distribution represents the job execution time pattern of a given task.
\end{itemize}

\section{Input and Output}
The simulator inputs are the following:
\begin{itemize}
    \item A fixed priority scheduling algorithm. As we will see in Section \ref{sec:simulator_implementation} the scheduling algorithm implemented is Deadline Monotonic: we can handle periodic and aperiodic tasks with a fixed priority.
    \item The taskset, characterized by the assumptions in Section~\ref{sec:simulator_task_model}.
    \item The duration of the simulation.
    \item The strategy for the deadline miss policy. We have the following option: 
    \begin{itemize}
        \item \textbf{Incomplete Jobs Aborted (JA)}: We abort the job that is responsable of the deadline miss.
        \item \textbf{Jobs run to Commpletion (JC)}: We contiue the simulation with the job that have excedded its deadline.
        \item There is also the policy where we abort teh whole simulation if we discover a deadline miss.
    \end{itemize}
\end{itemize}

All these aspect can be pass throght the \texttt{Main.hava} source file or by the \texttt{config.yaml} or \texttt{config.json} file. The second requires the code for the handling of the mapping from the config file to the Java object; these code is a simple call to the already implemented \texttt{ConfigLoader.java} class. This handling is made with the \textit{Jackson} library, that is responsable to the serialization of the configuration in Java objects.

\subsection{Output - Logging}
To test the scheduler algorithm implementation and to have evidence of the system behavior, the simulator can produce a file, named \textit{trace.log}. This file contains a sequence of \textit{<time, event>}, where \textit{event} can be: the release of a job, the execution and the completion of a job, the task preemption and the deadline miss event.

\subsection{Output - Distribution}
During the simulation the system retrieves all execution times sampled by all tasks using the \texttt{TaskExecutionTimeCollector} class. These data are sent to a Python script that is the only one responsible for dealing with them. The final output is a distribution of execution time for each task.

\subsection{Output - Deadline Miss}
As we mentioned before, the simulator can identify a deadline miss event. This doesn't implies the abort of the whole simulation, but only the abort fo the job that has missed its deadline; in this way the simulator can go on. In each case of the deadline miss we use an event collector that stores the number of aborted jobs on the number of released jobs, for each task. In this way at the end of the simulation we can extract a statistic aboud the aborted jobs.

\section{Implementation}
\label{sec:simulator_implementation}
In Figure~\ref{fig:simulator_class_diagram} is shown the class diagram of the simulator. In the following we present the core entities that compose the system.
\begin{itemize}
    \item \textbf{Client} \\
        Is the Main of the Java system. It is responsible for allocating the taskset and the scheduler, calling \texttt{analyze()} on the scheduler and the \texttt{callPythonExtractor} of the \texttt{MyUtils} class.
    \item \textbf{TaskSet} \\
        Is a simple set of all tasks.
    \item \textbf{Task} \\
        Represents a task. It models only the static specification of the task: the period, the execution time and the deadline; the first two are Sirio \texttt{Sampler} objects that represent the distribution from which they sample the true duration; the last is a \texttt{Duration} object, where the time must be expressed in milliseconds. Internally it also maintains a \texttt{jobCounter} to uniquely identify its jobs.
    \item \textbf{Job} \\
        Represents a task instance. While a Task contains only the task specifications, a Job contains the active state: the absolute deadline, the remaining execution time, the information about its completeness and the time of completion. It has a method to handle the simulation of its execution.
    \item \textbf{Scheduler} \\
        Represents the heart of the simulation, which uses the taskset for the scheduling simulation. It's responsible for initializing the system clock with a \texttt{MyClock} object and handling the system logging and the simulation duration.
    \item \textbf{FixedPriorityScheduler} \\
        Represents the abstraction for all fixed priority scheduling algorithms. All its subclasses must only implement the assign priority method.
    \item \textbf{DeadlineMonotonicScheduler} \\
        Is the specialization of the \texttt{FixedPriorityScheduler} that implements the priority assignment of the Deadline Monotonic scheduling algorithm.
    \item \textbf{Event} \\
        Is the abstract class that represents the relevant events of the system; each event is defined by the time of its occurrence. The idea is that the time can go forward until it reaches an instance of this class. The two implementations are: \texttt{ReleaseEvent} which models the job release event; \texttt{DeadlineEvent} which models the reaching of the job deadline.
    \item \textbf{DeadlineMissStrategy} \\
        Is the interface responsable to the implementation of the jobs deadliness missed policy. There are the possibility: the \texttt{AbortSimulationStrategy} that interrups the whole simulation; the \texttt{AbortJobStrategy} that abort the job and cuntinue the simulation; the \texttt{ContinueStrategy} that do nothing and leave the job in the ready job queque. All these three solution the job abortion and collect the related statistics.
\end{itemize}

\subsection{Utilities}
If we observe Figure~\ref{fig:simulator_class_diagram} we have to describe the system utilities.
\begin{itemize}
    \item \textbf{ConfigLoader} \\
        It is the class responsable to the mapping from the \texttt{config.yaml} file to the Java objects.
    \item \textbf{Sirio.Sampler} \\
        It is the Sirio interface. We have defined two implementation:
        \begin{itemize}
            \item \textbf{DeterministicSampler} \\
                It always returns a constant value. Usefull for deterministic parameters that allow also probabilistic values.
            \item \textbf{DiscreteChoiceSampler} \\
                Represent a weighted choice.
        \end{itemize}
    \item \textbf{MyClock} \\
        Represents the global clock of the system. Implements the method to go at a specified time, to go forward in time by a specified duration and to print the current time.
    \item \textbf{TraceLogger} \\
        The logging utils that is responsible to logging the scheduling main events in a file named \textit{trace.log} with a personalized format.An interface that abstracts the logging operations.
    \item \textbf{TaskExecutionTimeCollector} \\
        It is the collector of all execution times of all tasks.
    \item \textbf{DeadlineMissCollector} \\
        It is the collector of all aborted jobs of all tasks. This is useful to have a trace of all jobs that have been aborted due to deadline misses, to analyze them in a post-mortem way.
\end{itemize}

\subsection{Flow}
As illustrated in Figure~\ref{fig:simulator_activity_diagram}, the scheduler flow follows these steps:
\begin{enumerate}
    \item Assign task priorities according to the scheduling algorithm.
    \item Reset the system state. This allows the client (i.e., \texttt{Main.java}) to call \texttt{analyze()} multiple times for consecutive simulations on the same instance.
    \item Release the first job for each task in the taskset.
    \item Iterate through the events in the queue until the next event time exceeds the simulation duration.
    \begin{enumerate}
        \item Retrieve the first event from the queue and calculate the available time between the current clock and the event time.
        \item Distribute this available time among the ready jobs according to their priority.
        \begin{enumerate}
            \item Access the highest priority job from the ready queue.
            \item Check if the last executed job must be preempted by the current highest priority job for logging purposes.
            \item Execute the job for the available time. If the job finishes, remove it from the ready list.
            \item Advance the system clock by the actual execution time and check for deadline misses.
            \item Update the remaining available time.
        \end{enumerate}
        \item Synchronize the system clock with the event time and check for deadline misses for all ready jobs to ensure the state is consistent.
        \item Process all events that occur at the same time as the current clock. For a new release: create the job, sample its execution time, and add both its deadline and its next release time to the event queue.
    \end{enumerate}
    \item If the simulation duration has not been reached, execute the simulation for the remaining time.
\end{enumerate}

\section{Distribution Extraction}
\label{sec:simulator_distribution_extraction}
Since the main objective of the whole system is to produce the distribution of the tasks execution time, we have implemented a Python script that can be launch directly from \texttt{Main.java} with a java call, immediately after the generation of the simulation data. This prevents us from jumping from a Java system to a Python script: we only have to run the simulator system.

The Python script represents the information retrived during the simulation by the \texttt{TaskExecutionTimeCollector} collector in a \textit{.pdf} file in \textit{results/} directory.

\subsection{Java}
The Java \texttt{Main} creates a \texttt{ProcessBuilder} to execute directly the Python script and also redirects the console and the bash: now Python has full control.

Once the simulator has produced the data, they are stored in a \texttt{Map} object. Starting from this map, we use the \textit{Jackson} library to map data to a JSON file and transfer them through a pipe in an efficient way without writing any new code; the transfer is made through the Python script's standard input, to avoid the generation of temporary files that clutter the project file system.

\subsection{Python}
Once Python has read the whole data, it converts the JSON file into a dictionary. At this point, it is very simple to obtain the distribution of the execution time of all tasks.

\section{Testing}
The simulator tests are divided into two categories: the deterministic system test and the stochastic test.

\subsection{System Test}
This kind of test always uses the \texttt{DeterministicSampler}. The objective is to verify the correct behavior of the simulator system in various scheduling scenarios.

The scenarios that were tested are: single task execution; properties of the DM scheduling; task preemption (including nested preemption); overload of the system, i.e., execution of a taskset with a load factor greater than 1 resulting in aborted jobs; idempotency of the analysis process (i.e., multiple consecutive calls of \texttt{analyze()} produce the same result); offset management for tasks with non-zero initial release times; handling of priority collisions; boundary cases such as jobs completing exactly at their deadline or tasks with zero execution time; semantic validation of the traces; correct handling of all possible scenario on deadline miss discoverty.

These tests guarantee: the correctness of the DM scheduling algorithm; the precision of the clock implementation; the reliability of reusing the same instance for state handling; and the robustness of the engine in complex scheduling scenarios.

\subsection{Stochastic Test}
This kind of simulation uses real stochastic distributions to sample the parameters of the jobs.

The scenarios tested are the following: repeated simulations with different stochastic distributions (e.g., \texttt{UniformSampler} and \texttt{ExponentialSampler}) and multiple tasks; engine robustness with high task density and random loads to ensure simulation stability.
