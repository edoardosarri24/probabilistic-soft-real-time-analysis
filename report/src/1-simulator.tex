\chapter{Simulator}
The simulator's general objective is to have an instrument to validate our ideas and results in an experimental way. In particular we want to obtain experimentally for each task, the distribution of job execution time.

This kind of simulation must be totally deterministic; for this reason all timing aspects have been handled in a discrete way using the \texttt{Duration} Java type. In this way we can handle the complexity of small durations with a high abstraction level. We assume that the time is taken as input and returned as output in milliseconds. During the execution, to obtain the maximum precision, it's processed in nanoseconds.

\section{Taskset Model}
\label{sec:simulator_task_model}
In this section we define the taskset model, i.e., how the tasks in the taskset can be defined and what are their non-functional characteristics.
From now on we assume that:
\begin{itemize}
    \item We assume we have a single core CPU, which implies that we are working with a concurrent problem.
    \item For tasks:
    \begin{itemize}
        \item Each task is independent of the others. This means that there aren't shared resources among two tasks.
        \item Tasks can be both periodic and sporadic (i.e. the releases between two consecutive jobs of the same task are separated by a minimum inter-arrival time). For the sporadic task, each job release time can be sampled from some distribution.
        \item Each task has a single elementary execution unit, i.e. each task has only one chunk.
        \item The task execution time is defined probabilistically, i.e. the execution time of each job of a task is sampled from some distribution. These distributions can include also the deterministic ones.
        \item The deadline of each task is known a priori.
        \item The first release time is by deafult at time 0, but it ca be sampled from some distribution as well.
    \end{itemize}
    \item The job execution times are i.i.d. (i.e., independent and identically distributed). This means that only one distribution represents the job execution time pattern of a given task.
\end{itemize}

\section{Input and Output}
The simulator inputs are the following:
\begin{itemize}
    \item A fixed priority scheduling algorithm. As we will see in Section \ref{sec:simulator_implementation} the scheduling algorithm implemented is Deadline Monotonic: we can handle periodic and aperiodic tasks with a fixed priority.
    \item The taskset, characterized by the assumptions in Section~\ref{sec:simulator_task_model}.
    \item The duration of the simulation.
    \item The type of logging. By default no logging is performed.
\end{itemize}

\subsection{Output - Logging}
To test the scheduler algorithm implementation and to have evidence of the system behavior, the simulator can produce a file, named \textit{trace.log}, containing a sequence of \textit{<time, event>}, where \textit{event} can be: the release of a job, the execution and the completion of a job, the task preemption and the deadline miss event.

\subsection{Output - Deadline Miss}
The system has obviously the capabilities to identify a deadline miss by a task and can also identify the job that is responsible for this violation. A violation implies the abort of the whole simulation.

\subsection{Output - Distribution}
During the simulation the system retrieves all execution times sampled by all tasks using the \texttt{TaskExecutionTimeCollector} class. These data are sent to a Python script that is the only one responsible for dealing with them. The final output is a distribution of execution time for each task.

\section{Implementation}
\label{sec:simulator_implementation}
In Figure~\ref{fig:simulator_class_diagram} is shown the class diagram of the simulator. In the following we present the core entities that compose the system.
\begin{itemize}
    \item\textbf{Client} \\
        Is the Main of the Java system. It is responsible for allocating the taskset and the scheduler and calling \texttt{analyze()} on the scheduler.
    \item\textbf{TaskSet} \\
        Is a simple set of all tasks.
    \item\textbf{Task} \\
        Represents a task. It models only the static specification of the task: the period, the execution time and the deadline; the first two are Sirio \texttt{Sampler} objects that represent the distribution from which they sample the true duration; the last is a \texttt{Duration} object, where the time must be expressed in milliseconds. Internally it also maintains a \texttt{jobCounter} to uniquely identify its jobs.
    \item\textbf{Job} \\
        Represents a task instance. While a Task contains only the task specifications, a Job contains the active state: the absolute deadline, the remaining execution time, the information about its completeness and the time of completion. It has a method to handle the simulation of its execution.
    \item\textbf{Scheduler} \\
        Represents the heart of the simulation, which uses the taskset for the scheduling simulation. It's responsible for initializing the system clock with a \texttt{MyClock} object and handling the system logging and the simulation duration.
    \item\textbf{FixedPriorityScheduler} \\
        Represents the abstraction for all fixed priority scheduling algorithms. All its subclasses must only implement the assign priority method.
    \item\textbf{DMScheduler} \\
        Is the specialization of the \texttt{Scheduler} that implements the priority assignment of the Deadline Monotonic scheduling algorithm.
    \item\textbf{Event} \\
        Is the abstract class that represents the relevant events of the system; each event is defined by the time of its occurrence. The idea is that the time can go forward until it reaches an instance of this class. The two implementations are: \texttt{ReleaseEvent} which models the job release event; \texttt{DeadlineEvent} which models the reaching of the job deadline.
\end{itemize}

\subsection{Utilities}
If we observe Figure~\ref{fig:simulator_class_diagram} we have to describe the system utilities.
\begin{itemize}
    \item\textbf{DeterministicSampler} \\ A specialization of the Sirio \texttt{Sampler} interface that always returns a constant value, used for deterministic parameters.
    \item\textbf{MyClock} \\ Represents the global clock of the system. Implements the method to go at a specified time, to go forward in time by a specified duration and to print the current time.
    \item\textbf{MyLogger} \\ An interface that abstracts the logging operations.
    \item\textbf{TraceLogger} \\ Implementation of \texttt{MyLogger} that is responsible to logging the scheduling main events in a file named \textit{trace.log} with a personalized format.
    \item\textbf{NoLogger} \\ Implementation of \texttt{MyLogger} that doesn't perform any logging operation. It's the default logger if Main passes no logger.
    \item\textbf{DeadlineMissedException} \\ A custom exception thrown when a task's job exceeds its deadline.
    \item\textbf{TaskExecutionTimeCollector} \\ Is the collector of all execution times of all tasks.
\end{itemize}

\subsection{Flow}
Let's analyze, thanks also to Figure~\ref{fig:simulator_activity_diagram}, the scheduler flow. This follows the steps below:
\begin{enumerate}
    \item Assign priority according to the scheduling algorithm policy.
    \item Reset the state of the whole system. This allows the client (i.e., the \texttt{Main.java}) to call \texttt{analyze()} multiple times to obtain consecutive schedulings.
    \item Release the first job of all tasks.
    \item Iterate through all events until we find a next event with a time higher than the simulation duration.
    \begin{enumerate}
        \item Take and remove the first event from the event queue and calculate the available time from now to this current event.
        \item Distribute the available time among all ready jobs according to their priority.
        \begin{enumerate}
            \item Take the highest priority job from the ready jobs queue.
            \item Evaluate if the last job executed must be preempted by this new highest priority job.
            \item Give to the job the available time that it can exploit and execute the job.
            \item Advance the system clock by the real job execution time and check for deadline misses.
            \item Calculate the new available time.
            \item If the job hasn't finished its work we insert it again in the ready job list for the next iteration.
        \end{enumerate}
        \item Carry the logical clock current time to our event time. Then check for deadline misses for all active jobs; with this we ensure that the simulator state is correct and we can proceed.
        \item Consider all events that have the same time as our current event. We have to deal with them all together. For these events that represent a new release, we release a new job of the related task. The release behavior is: create a new job and add it to all related structures; trace the job execution time; add its deadline event to the event queue; add the new task release time to the event queue.
    \end{enumerate}
    \item If the current time is lower than the simulation duration, execute the simulation for the remaining time.
\end{enumerate}

\section{Testing}
The simulator tests are divided into two categories: the deterministic system test and the stochastic test.

\subsection{System Test}
This kind of test always uses the \texttt{DeterministicSampler}. The objective is to verify the correct behavior of the simulator system.

The scenarios that were tested are: single task execution; properties of the DM scheduling; task preemption; overload of the system, i.e. execution of a taskset with a load factor greater than 1; multiple consecutive calls of \texttt{analyze()} produce the same result; semantic validation of the traces; numerical stability with $10^6$ jobs released.

This huge number of tests guarantees: the correctness of the DM scheduling algorithm; the precision of the clock implementation; the reliability of reusing the same instance for state handling.

\subsection{Stochastic Test}
This kind of simulation uses real stochastic distributions to sample the parameters of the jobs.

The scenarios tested are the following: repeated simulations with different distributions and different tasks; we have stressed the queue management with a huge number of events.

\section{Distribution Extraction}
\label{sec:simulator_distribution_extraction}
Since the main objective of the whole system is to produce the distribution of the tasks' execution time, the Python script launch is made in \texttt{Main.java}, immediately after the generation of the simulation data. This prevents us from jumping from a Java system to a Python script: we only have to run the simulator system.

\subsection{Java}
The Java \texttt{Main} creates a \texttt{ProcessBuilder} to execute directly the Python script and also redirects the console and the bash: now Python has full control.

Once the simulator has produced the data, they are stored in a \texttt{Map} object. Starting from this map, we use the \textit{Jackson} library to map data to a JSON file and transfer them through a pipe in an efficient way without writing any new code; the transfer is made through the Python script's standard input, to avoid the generation of temporary files that clutter the project file system.

\subsection{Python}
Once Python has read the whole data, it converts the JSON file into a dictionary. At this point, it is very simple to obtain the distribution of the execution time of all tasks.
